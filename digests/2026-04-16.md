## Deep Dive of the Day

**Topic: The Three-Layer Agent Architecture — Model, Harness, Context.** This is the most interview-leverageable concept in today's feed. 
Model: the model weights themselves. Harness: the harness around the model that powers all instances of the agent. This refers to the code that drives the agent, as well as any instructions or tools that are always part of the harness. Context: additional context (instructions, skills) that lives outside the harness, and can be used to configure it.
 It matters because FAANG agent interviews increasingly probe *where* to place a capability: fine-tuning (model), system-prompt/tools/middleware (harness), or dynamic memory/skills (context) — each has different cost, latency, versioning, and multi-tenancy implications. Directions worth digging deeper: (1) harness optimization research (e.g., Meta-Harness-style end-to-end trace-driven optimization of system prompts and tools); (2) context/memory layering patterns — per-agent, per-tenant, per-user (OpenClaw-style `SOUL.md`, claude-mem compression, PageIndex vectorless retrieval); (3) the interaction with OBO identity and MCP governance as shown by Databricks' Unity AI Gateway, since harness-level tool calls must propagate user permissions cleanly. Being able to whiteboard a system across these three layers — and defend *why* a given feature lives where it lives — is the single best signal you can give in a FAANG agent-engineer loop.
