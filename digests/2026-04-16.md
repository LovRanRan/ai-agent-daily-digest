## 3. Engineering Practice / Technical Blogs

- **[LangChain — Deep Agents Deploy](https://www.langchain.com/blog/deep-agents-deploy-an-open-alternative-to-claude-managed-agents)** (April 15). 
"Harness engineering" has risen as the discipline of building harnesses to turn LLMs into agents. These harnesses contain orchestration logic, tools, and skills that serve as the foundation of the agent, while allowing builders to provide custom instructions, tools, and skills. To go to production requires deploying agent orchestration logic and memory in a multi-tenant scalable way, setting up sandboxes per agent session, and standing up endpoints from MCP to A2A for human-in-the-loop and memory — all bundled into a single `deepagents deploy` command.
 Takeaway: concrete open-source answer to Claude Managed Agents; read it for the orchestration/sandbox/endpoint decomposition alone.

- **[SkillsBench Paper Review — AI with Mike](https://aiwithmike.substack.com/p/anthropics-revolution-on-trial-do)** (April 15). 
In Software Engineering, the improvement from skills was negligible (<4.5%) — models already know how to code. When models were asked to generate their own skills, performance dropped by 1.3%. Using four or more skills dropped the improvement to just 5.9pp (compared to ~18pp with 2-3 skills); too many instructions create confusion and "noise." "Comprehensive" skills actually hurt performance by 2.9pp — an agent needs a checklist, not an encyclopedia. Notably, Claude Haiku 4.5 with skills (27.7%) outperformed Claude Opus 4.5 without skills (22.0%): a good skill can compensate for a weaker model in specific tasks.
 Takeaway: hard data that context is a constrained resource — more skills ≠ better agent; the sweet spot is 2–3 concise skills.

- **[Anthropic Engineering — Equipping agents with Agent Skills](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills)** (still the canonical reference). 
Progressive disclosure is the core design principle that makes Agent Skills flexible and scalable. Like a well-organized manual that starts with a table of contents, then specific chapters, and finally a detailed appendix, skills let Claude load information only as needed — agents with a filesystem and code execution tools don't need to read the entirety of a skill into their context window when working on a particular task. This means that the amount of context that can be bundled into a skill is effectively unbounded.
 Takeaway: the SKILL.md + progressive disclosure pattern is now industry-standard context engineering — 
OpenAI has quietly adopted structurally identical architecture in both ChatGPT and its Codex CLI tool, with the same file naming conventions, the same metadata format, and the same directory organization.


- **[Anthropic Engineering — Infrastructure effects on agentic coding benchmarks](https://www.anthropic.com/engineering)**.

 Takeaway: evaluation hygiene is an under-appreciated source of variance — critical reading before you cite any agent benchmark in a design doc.

## Deep Dive of the Day

**Progressive disclosure and Agent Skills as context engineering.** For a FAANG AI agent engineer interview, the single most valuable thread today is the convergence around Skills + progressive disclosure. The principle is that 
skills are triggered in the context window via the system prompt — the context window starts with the core system prompt and metadata for each installed skill along with the user's initial message, and Claude triggers deeper content only when relevant.
 This matters because it reframes the "bigger context window" arms race: 
simply increasing context windows does not equate to stronger agents — when context exceeds 100k tokens, reasoning costs skyrocket while critical instructions often get lost amid noise.
 SkillsBench's empirical result — that 2–3 focused skills beat 4+ skills and that a small model with skills can beat a frontier model without — gives you a quantitative answer to the classic interview question "how would you make this agent cheaper and more reliable?" Dig deeper into: (1) the SKILL.md spec and how OpenAI/Microsoft/Figma are adopting it, (2) the relationship between Skills (procedural knowledge) and MCP (tool connectivity) as complementary layers, and (3) context-window economics — tokens-per-turn vs. effort parameters in Opus 4.7, where the same input can map to 1.0–1.35× more tokens depending on content type.
