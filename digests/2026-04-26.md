## 3. Engineering Practice / Technical Blogs

- **[Anthropic Engineering — Managed Agents / harness virtualization](https://www.anthropic.com/engineering/managed-agents)** (via Microsoft analysis). 
While building the agent scaffolding (the "harness") for Claude Sonnet 4.5, they noticed the model suffered from "context anxiety," so they added context-reset logic into the harness. But when the same harness ran on the more capable Claude Opus 4.5, those resets became dead weight — the stronger model no longer needed them, yet the harness was actively holding it back. This is the fundamental dilemma of the harness: it encodes assumptions about the current model's capabilities, and those assumptions rot quickly as models evolve
. Takeaway: 
Managed Agents virtualizes the Agent runtime into three independent interface layers — Brain / Hands / Session — which map to reasoning orchestrator / execution sandbox / durable event log
.

- **[Addy Osmani — Agent Harness Engineering](https://addyosmani.com/blog/agent-harness-engineering/)**. 
The harness is the prompts, tools, context policies, hooks, sandboxes, subagents, feedback loops, and recovery paths wrapped around the model so it can actually finish something. A decent model with a great harness beats a great model with a bad harness
. Concrete techniques: 
Compaction — when the window gets close to full, the harness intelligently summarizes and offloads older context. Tool-call offloading — large tool outputs clutter context; the harness keeps the head and tail tokens above a threshold and offloads the full output to the filesystem. Skills with progressive disclosure — loading every tool and MCP into context at startup degrades performance, so skills let the harness reveal instructions and tools only when the task actually calls for them
.

- **[HumanLayer — Skill Issue: Harness Engineering for Coding Agents](https://www.humanlayer.dev/blog/skill-issue-harness-engineering-for-coding-agents)**. The most useful debunk of the year: 
Sub-agents are a popular but often misunderstood harness configuration point. The "frontend engineer" sub-agent and "backend engineer" sub-agent thing doesn't work. What does work is using sub-agents for context control. They provide a way to encapsulate an entire coding agent session's worth of work such that the dispatching agent only sees the prompt it writes for the sub-agent, and the sub-agent's final result. None of the intermediate tool calls, tool results, or other messages end up in the parent coding agent's context window. Breaking work up into discrete tasks and delegating it to sub-agents is how we keep our primary coding agent thread in the "smart zone."


- **[Daily Dose of DS — The Anatomy of an Agent Harness](https://blog.dailydoseofds.com/p/the-anatomy-of-an-agent-harness)**. The mental model worth memorizing: 
A raw LLM is a CPU with no RAM, no disk, and no I/O. The context window serves as RAM (fast but limited). External databases function as disk storage (large but slow). Tool integrations act as device drivers. The harness is the operating system
. 
Two products using identical models can have wildly different performance based solely on harness design. The TerminalBench evidence is clear that changing only the harness moved agents by 20+ ranking positions
.

- **[LangChain Blog — Deep Agents](https://www.langchain.com/blog/deep-agents)** (refreshed thread). 
Using an LLM to call tools in a loop is the simplest form of an agent. This architecture, however, can yield agents that are "shallow" and fail to plan and act over longer, more complex tasks. Applications like "Deep Research", "Manus", and "Claude Code" have gotten around this limitation by implementing a combination of four things: a planning tool, sub agents, access to a file system, and a detailed prompt
. The four-component recipe is a near-universal pattern for FAANG-grade agent design.

## Deep Dive of the Day

**Topic: Harness Engineering — and specifically the principle that harnesses encode assumptions that rot as models improve.**

This is the single most interview-relevant idea today because it reframes an "AI agent engineer" as primarily a systems designer, not a prompt author. 
The "agent" is the emergent behavior — the goal-directed, tool-using, self-correcting entity the user interacts with. The harness is the machinery producing that behavior. When someone says "I built an agent," they mean they built a harness and pointed it at a model
. Anthropic's Managed Agents post crystallized why this matters: 
harnesses encode assumptions about what Claude can't do on its own; those assumptions need to be frequently questioned because they can go stale as models improve. As just one example, in prior work Claude Sonnet 4.5 would wrap up tasks prematurely as it sensed its context limit approaching — a behavior sometimes called "context anxiety." This was addressed by adding context resets to the harness
 — but those same resets later hurt Opus 4.5.

Directions worth digging deeper into for a FAANG interview: (1) the **eleven-component anatomy** (agent loop, context assembly, tool dispatch, memory, sub-agent context firewalls, permission gates, sandboxes, hooks, verification, compaction, lifecycle); (2) the **Brain/Hands/Session virtualization** pattern from Anthropic Managed Agents — be ready to whiteboard a durable event log + execution sandbox + reasoning orchestrator split; (3) **context engineering tactics** — tool-call offloading, progressive skill disclosure, context resets, sub-agents as context firewalls; (4) the **TerminalBench result that harness design alone moves agents 20+ ranking positions** — a perfect data point to cite when an interviewer asks "model choice or system design?". Practice articulating the Beren Millidge OS analogy and one concrete failure mode you'd debug (e.g., the Claude Code caching/extended-thinking bug) — that combination of vocabulary and war-story specifics is what distinguishes a strong agent-engineer candidate.
