- **[Anthropic Engineering — Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents).** Core takeaway: 
The core challenge of long-running agents is that they must work in discrete sessions, each starting with no memory — like a project staffed by engineers in shifts who arrive with no memory of the previous shift
. 
The key insight is finding a way for agents to quickly understand the state of work when starting with a fresh context window, achieved with a claude-progress.txt file alongside the git history
.

- **[AddyOsmani — Agent Harness Engineering](https://addyosmani.com/blog/agent-harness-engineering/).** Core takeaway: 
"Agent = Model + Harness. If you're not the model, you're the harness." A harness is every piece of code, configuration, and execution logic that isn't the model itself; a raw model is not an agent
. 
Planner / generator / evaluator splits — Anthropic's long-running harness work is explicit that separating generation from evaluation into distinct agents outperforms self-evaluation, because agents reliably skew positive when grading their own work
.

- **[HumanLayer — Skill Issue: Harness Engineering for Coding Agents](https://www.humanlayer.dev/blog/skill-issue-harness-engineering-for-coding-agents).** Core takeaway: 
Stuffing every instruction and tool into the system prompt makes agents worse — you blow through your instruction budget before the agent starts. Skills solve this through progressive disclosure: the agent only gets specific instructions or tools when it (or you) decide they're needed
. 
If an MCP server duplicates a CLI well-represented in training data, prompt the agent to use the CLI instead — the model already knows how, and you gain composability with grep/jq
.

- **[Epsilla — The GAN-Style Agent Loop: Anthropic's Harness Architecture](https://www.epsilla.com/blogs/anthropic-harness-engineering-multi-agent-gan-architecture).** Core takeaway: 
The Evaluator, armed with Playwright MCP, acts as a QA engineer testing UI, API endpoints, and database state against a sprint contract negotiated with the Generator before any code is written. Tasked with creating a 2D retro game-making tool, a single agent produced a barely-functional prototype in 20 minutes for $9; the full harness ran for 6 hours, cost $200, and delivered a polished application — a phase change in capability
.

## Deep Dive of the Day

**Topic: Agent Harness Engineering and the Generator/Evaluator (GAN-style) split.**

This is the single most valuable pattern to internalize for a FAANG agent-engineer interview right now because the entire frontier — Anthropic's Managed Agents, OpenAI's Codex, Google's Interactions API — is converging on the same abstraction: **agent = model + harness**, where the harness owns sessions, sandboxes, hooks, skills, and subagent orchestration. 
The planner/generator/evaluator split outperforms self-evaluation because agents reliably skew positive when grading their own work
, and 
grounding the evaluator with tools like Playwright MCP and a pre-negotiated "sprint contract" produces a phase change — not a marginal improvement — in long-horizon task quality
. Worth digging deeper into: (1) **context-bridging artifacts** like progress files plus git history that let stateless sessions resume coherently (
the claude-progress.txt + git history pattern
); (2) **progressive disclosure of skills/tools** to avoid blowing the instruction budget (
skills loaded only when the agent decides they're needed
); (3) **prefix-caching sensitivity** (
dynamic prompt order changes cause 3-5x latency spikes — keep static core first
); and (4) the architectural question of **what should be model-native vs. harness-supplied** as models improve, since harness components encode assumptions about model weakness that go stale fast. Expect interview probes on designing a multi-context-window coding agent, choosing between compaction vs. context resets, and justifying when to use MCP vs. plain CLI tools.
